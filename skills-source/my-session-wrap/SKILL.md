# My Session Wrap

세션을 마무리할 때 작업 내용을 정리하고, 문서를 업데이트하고, 배운 것을 기록하는 스킬.

## Execution Flow

```
┌─────────────────────────────────────────────────┐
│  Step 1: Git 상태 확인                            │
│  (변경/생성/삭제된 파일 목록 파악)                   │
├─────────────────────────────────────────────────┤
│  Step 2: Phase 1 — 4개 분석 에이전트 (병렬)        │
│     ┌──────────────┬──────────────┐              │
│     │ doc-updater  │ automation-  │              │
│     │ 문서 업데이트  │ scout        │              │
│     │ 필요한 곳 탐색 │ 자동화 제안   │              │
│     ├──────────────┼──────────────┤              │
│     │ learning-    │ followup-    │              │
│     │ extractor    │ suggester    │              │
│     │ 배운 것 정리  │ 다음 할 일    │              │
│     └──────────────┴──────────────┘              │
├─────────────────────────────────────────────────┤
│  Step 3: Phase 2 — 검증 에이전트 (순차)            │
│     duplicate-checker: 중복 제거 + 병합            │
├─────────────────────────────────────────────────┤
│  Step 4: 결과 통합                                │
│  (4개 분석 + 검증 결과를 하나로 정리)               │
├─────────────────────────────────────────────────┤
│  Step 5: 사용자 선택 + 실행                        │
│  (AskUserQuestion으로 실행할 작업 선택)             │
└─────────────────────────────────────────────────┘
```

## Step 1: 변경 내용 확인

이 세션에서 어떤 작업을 했는지 자동으로 파악한다.

### 수집 방법

먼저 `git status`를 실행하여 git 저장소인지 확인한다.

**Git이 있는 경우:**
1. `git status`로 변경/생성/삭제된 파일 목록을 확인한다
2. `git diff --stat`으로 수정된 파일의 변경 규모를 파악한다
3. 새로 생성된 파일(untracked)도 포함한다

**Git이 없는 경우 (fallback):**
1. 이번 대화에서 Write/Edit 도구로 생성하거나 수정한 파일 목록을 대화 컨텍스트에서 추출한다
2. git 관련 에러를 1회만 확인하고, 실패하면 즉시 fallback으로 전환한다 (반복 시도 금지)

### 출력 형식

변경 내용을 아래 형식으로 요약한다:

```
이번 세션 변경 사항:
- 새로 생성: 3개 파일 (SKILL.md, config.json, ...)
- 수정: 2개 파일 (CLAUDE.md, README.md)
- 삭제: 0개 파일
- Git 상태: 사용 가능 / 미초기화
```

이 요약을 Phase 1의 4개 에이전트에게 컨텍스트로 전달한다.

## Step 2: Phase 1 — 4개 분석 에이전트 (병렬)

Step 1에서 파악한 변경 내용을 4개 전문가 에이전트에게 동시에 전달한다.
4개 에이전트는 서로 의존하지 않으므로 **병렬로 동시 실행**한다.

### 에이전트 공통 규칙

모든 에이전트에게 아래 규칙을 프롬프트에 반드시 포함한다:

1. **이번 세션 범위만 분석한다.** Step 1에서 전달받은 변경 파일과 작업 내용만 다룬다. 이전 세션이나 프로젝트 전체를 분석하지 않는다.
2. **간결하게 출력한다.** 각 항목은 핵심 한 줄 + (필요시) 2~3줄 설명으로 제한한다. 변경 사항이 적은 세션에서는 출력도 짧아야 한다.
3. **제안이 없으면 "없음"으로 출력한다.** 억지로 제안을 만들어내지 않는다.

### Agent 1: doc-updater (문서 업데이트 탐색)

변경된 파일을 기반으로 업데이트가 필요한 문서를 찾는다.

- CLAUDE.md에 새로운 스킬/설정을 반영해야 하는지 확인
- README.md에 변경 사항을 기록해야 하는지 확인
- 기존 문서와 실제 코드 사이의 불일치를 탐색

출력: 업데이트가 필요한 문서 목록 + 구체적 변경 제안 (각 제안은 3줄 이내)

### Agent 2: automation-scout (자동화 기회 발견)

이번 세션의 작업 패턴에서 자동화할 수 있는 부분을 찾는다.

- 반복적으로 수행한 작업이 있는지 분석
- 스킬이나 스크립트로 만들 수 있는 패턴 식별
- 기존 스킬로 대체할 수 있었던 수동 작업 발견

출력: 자동화 제안 최대 3개 (우선순위순, 각 제안은 3줄 이내)

### Agent 3: learning-extractor (학습 내용 정리)

이번 세션에서 배운 것을 정리한다. 이전 세션의 학습은 포함하지 않는다.

- 새로 알게 된 기술, 도구, 패턴
- 시행착오를 통해 얻은 교훈
- 다른 프로젝트에도 적용할 수 있는 인사이트

출력: 학습 항목 최대 5개 (핵심 한 줄 + 1~2줄 설명)

### Agent 4: followup-suggester (후속 작업 제안)

다음 세션에서 이어서 해야 할 작업을 제안한다.

- 이번 세션에서 미완성인 작업
- TODO/FIXME 등 남겨둔 표시 수집
- 자연스럽게 이어지는 다음 단계 제안

출력: 후속 작업 최대 5개 (긴급도/중요도 표시, 각 항목은 2줄 이내)

## Step 3: Phase 2 — 검증 에이전트 (순차)

Phase 1의 4개 에이전트 결과를 모두 받은 후, 순차적으로 검증한다.
Phase 1 결과가 모두 완료되어야 실행할 수 있으므로 **Phase 1 이후에 순차 실행**한다.

### duplicate-checker (중복 검증)

Phase 1에서 나온 모든 제안을 비교하여 중복을 처리한다.

**검증 기준:**

| 유형 | 처리 방법 | 예시 |
|------|----------|------|
| 완전 중복 | 하나만 남기고 제거 | doc-updater와 followup-suggester가 둘 다 "CLAUDE.md 업데이트" 제안 |
| 부분 중복 | 하나로 병합 제안 | "README 수정"과 "README에 스킬 추가" → "README에 새 스킬 정보 추가"로 병합 |
| 충돌 | 두 제안을 함께 표시하고 사용자 판단 요청 | 한쪽은 "파일 삭제" 다른 쪽은 "파일 수정" 제안 |

출력: 중복 제거/병합된 최종 제안 목록

## Step 4: 결과 통합

Phase 1(4개 에이전트) + Phase 2(검증) 결과를 하나로 합쳐서 사용자에게 보여준다.

### 통합 보고서 형식

```
## 세션 마무리 보고서

### 변경 요약
- 새로 생성: N개 파일
- 수정: N개 파일

### 문서 업데이트 제안 (doc-updater)
- [ ] CLAUDE.md에 새 스킬 정보 추가
- [ ] README.md 업데이트

### 자동화 기회 (automation-scout)
- [ ] 반복 패턴 → 스킬로 전환 가능

### 오늘 배운 것 (learning-extractor)
- 핵심 학습 1
- 핵심 학습 2

### 다음 할 일 (followup-suggester)
- [ ] 우선순위 높음: ...
- [ ] 우선순위 보통: ...

### 검증 노트 (duplicate-checker)
- 중복 제거: N건
- 병합: N건
```

## Step 5: 사용자 선택 + 실행

통합 보고서를 보여준 후, AskUserQuestion으로 실행할 작업을 선택받는다.

### 선택지

AskUserQuestion으로 아래 선택지를 제시한다.
**단, Git이 초기화되지 않은 경우 "커밋하기"는 선택지에서 제외한다.**

**Git이 있는 경우:**

| 선택지 | 설명 |
|--------|------|
| 커밋하기 | 변경 사항을 git commit으로 저장 |
| 문서 업데이트 실행 | doc-updater가 제안한 문서 수정을 실행 |
| 전부 실행 | 커밋 + 문서 업데이트 + 학습 기록 저장 |
| 건너뛰기 | 보고서만 확인하고 종료 |

**Git이 없는 경우:**

| 선택지 | 설명 |
|--------|------|
| 문서 업데이트 실행 | doc-updater가 제안한 문서 수정을 실행 |
| 전부 실행 | 문서 업데이트 + 학습 기록 저장 (커밋 제외) |
| 건너뛰기 | 보고서만 확인하고 종료 |

### 실행

사용자가 선택한 작업만 실행한다:

- **커밋하기** (Git 있을 때만): 변경된 파일을 stage하고 세션 요약을 커밋 메시지로 사용
- **문서 업데이트**: doc-updater가 제안한 변경을 실제로 적용
- **전부 실행**: 위 항목 모두 + 학습 기록을 `learnings/YYYY-MM-DD-session.md`에 저장
- **건너뛰기**: 아무것도 실행하지 않고 종료

### 학습 기록 저장 규칙

- 저장 경로: `learnings/YYYY-MM-DD-session.md` (프로젝트 루트 기준)
- 같은 날 여러 세션이 있으면: `learnings/YYYY-MM-DD-session-2.md`
- 내용: 세션 요약 + 배운 것(learning-extractor 결과) + 다음 할 일(followup-suggester 결과)
